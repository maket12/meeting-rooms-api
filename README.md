Room Booking Service
Сервис для управления переговорками и их бронирования. Позволяет администраторам настраивать расписание доступности ресурсов, а пользователям — находить свободные слоты и создавать брони.

🛠 Технологический стек
Язык: Go (чистый net/http без сторонних фреймворков для минимизации зависимостей).

База данных: PostgreSQL.

Авторизация: JWT (с реализованным TTL для безопасности).

Транзакции: avito-tech/go-transaction-manager для обеспечения атомарности бизнес-логики.

Миграции: golang-migrate.

Контейнеризация: Docker Compose.

🏛 Принятые архитектурные решения
1. Генерация слотов (Lazy Loading)
   Согласно ТЗ, 99.9% запросов приходятся на ближайшие 7 дней. Мы выбрали подход ленивой генерации (on-demand):

Слоты не генерируются на годы вперед.

При запросе списка слотов на конкретную дату система проверяет их наличие в БД.

Если слотов нет, они мгновенно нарезаются на основе расписания и сохраняются в БД.

Обоснование: Это позволяет поддерживать производительность при RPS 100 и SLI 200мс, не раздувая базу данных миллионами пустых записей.

2. Детерминированные UUID для слотов
   Для предотвращения дубликатов при одновременных запросах (Race Condition) используются детерминированные UUID. UUID слота генерируется на основе комбинации room_id, date и времени начала. Это гарантирует идемпотентность вставки слотов в БД.

3. Ограничения времени (30 минут)
   В расписании минуты ограничены значениями 00 и 30.

Обоснование: Так как длительность слота фиксирована (30 мин), это позволяет поддерживать «чистую» временную сетку и упрощает генерацию стабильных идентификаторов.

4. Работа с внешним Conference Service
   Реализована имитация интеграции с внешним сервисом ссылок. Приняты следующие правила:

Отказоустойчивость: Если пользователь запросил ссылку, но внешний сервис недоступен — транзакция создания брони откатывается (rollback). Мы считаем, что создание брони без обещанной ссылки — это нарушение бизнес-ожиданий пользователя.

5. Безопасность и фильтрация
   JWT: user_id и role извлекаются строго из токена.

Фильтрация /bookings/my: Реализована на уровне SQL с использованием NOW() AT TIME ZONE 'UTC'. Пользователь видит только актуальные будущие брони.

Отмена брони: Сделана идемпотентной. Повторный вызов на отмененной брони возвращает 200 OK.
